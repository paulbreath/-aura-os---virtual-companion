# /office-hours - YC风格产品诊断

在开始任何新功能之前，进行产品诊断。

## 触发条件
- 用户描述新功能需求
- "我想添加..."、"能不能做..."、"需要一个..."

## 工作流程

### 1. 痛点挖掘 (5个问题)
不要直接问"你想要什么功能"，而是问：
1. **具体场景**: "上次遇到这个问题是什么时候？能描述当时的具体情况吗？"
2. **当前解决方案**: "现在你是怎么处理的？"
3. **成本**: "这个问题让你多花了多少时间/钱？"
4. **频率**: "多久遇到一次？"
5. **后果**: "不解决会怎样？"

### 2. 挑战假设 (Push Back)
不要直接接受用户的功能描述，而是挑战：
- "你确定需要这个功能，还是只是觉得需要？"
- "能不能用更简单的方式解决？"
- "这个功能的核心价值是什么？"

### 3. 方案生成
生成 2-3 个实现方案：
- **MVP**: 最小可行产品，1天内完成
- **标准**: 完整功能，1周内完成  
- **高级**: 带优化的完整方案

### 4. 输出格式

```
## 🎯 产品诊断

### 痛点分析
- 场景: [用户描述的具体场景]
- 成本: [时间/金钱成本]
- 频率: [频率]

### 方案对比
| 方案 | 时间 | 复杂度 | 价值 |
|------|------|--------|------|
| MVP | 2小时 | 低 | 验证核心需求 |
| 标准 | 1天 | 中 | 完整功能 |
| 高级 | 3天 | 高 | 最佳体验 |

### 建议
[推荐方案和理由]

### 下一步
[具体的实现计划]
```

## 示例

用户: "我想要一个视频生成功能"

好的回应:
1. 先问："上次你想给角色生成视频是什么情况？具体想做什么？"
2. 挑战："你确定需要视频，还是动态图片就够了？"
3. 提供方案：MVP(静态GIF)、标准(5秒视频)、高级(可交互视频)
